# PROJECT: SV5T (Sinh Viên 5 Tốt) Management System - HANU

## 1. TECH STACK & ARCHITECTURE
- **Backend:** Java, Spring Boot 3.x, Spring Data JPA, Hibernate, Spring Security.
- **Database:** MySQL.
- **Frontend:** NuxtJS (Port 3000).
- **Architecture Pattern:** Monolith, RESTful API, Stateless (JWT), Backend For Frontend (BFF) pattern cho việc tổng hợp dữ liệu (Data Aggregation).
- **Scope:** ~100 concurrent users

## 2. CORE DOMAIN LOGIC
### A. Authentication
- **Strict Email Policy:** Chỉ chấp nhận đăng ký từ domain `@ms.hanu.edu.vn`.
- **Verification:** Xác thực thủ công bằng OTP qua Email (OutLook). TUYỆT ĐỐI KHÔNG dùng OAuth2 do rào cản chính sách của trường.
- **Password Reset Flow:** Link đổi mật khẩu gửi qua mail phải trỏ về UI của Frontend. Backend chỉ cung cấp API `GET /verify_token` và `POST /reset_password` (nhận JSON Body). Không xử lý UI redirect ở Backend.

### B. Master Data (Hệ thống Tiêu chí động)
- **Hierarchy:** `Campaign` (Đợt xét) -> `Standard` (Tiêu chuẩn) -> `Criteria` (Tiêu chí - Cấu trúc cây).
- **Tree Structure:** Bảng `Criteria` dùng Adjacency List pattern (`parent_id` tự tham chiếu). 
- **Dynamic Rules:** Logic "Đạt 1 trong các tiêu chí sau" được giải quyết bằng cột `is_mandatory` (Boolean) và `required_children_count` (Integer).
- **Evidence Type:** Lưu dưới DB là `VARCHAR(50)`, nhưng trong Java Entity phải map với Enum bằng `@Enumerated(EnumType.STRING)`.

### C. Transaction Data (Quản lý Hồ sơ)
- **Application:** Mỗi sinh viên (`user_id`) chỉ có 1 hồ sơ trong 1 Đợt xét (`campaign_id`). Trạng thái: `DRAFT`, `SUBMITTED`, `APPROVED`, `REJECTED`.
- **Evidence:** Lưu trữ minh chứng cho từng Tiêu chí là lá (Leaf Criteria). Chỉ lưu URL Cloud (S3/Cloudinary), KHÔNG LƯU BLOB file dưới DB.

## 3. STRICT CODING STANDARDS (NEVER VIOLATE)
### A. JPA & Hibernate (Critical)
- **NO `@Data` on Entities:** Gây lỗi đệ quy vô hạn. Chỉ dùng `@Getter`, `@Setter`, `@Builder`, `@NoArgsConstructor`, `@AllArgsConstructor`.
- **Prevent Infinite Recursion:** Bắt buộc có `@EqualsAndHashCode.Exclude` và `@ToString.Exclude` trên TẤT CẢ các field quan hệ (`@OneToMany`, `@ManyToOne`).
- **Handle Proxy Serialization:** Thêm `@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})` trên tất cả Entity để tránh lỗi Jackson khi load LAZY.
- **Solve `MultipleBagFetchException`:** BẮT BUỘC dùng `Set<T>` (ưu tiên `LinkedHashSet` để giữ thứ tự) thay vì `List<T>` cho các quan hệ `@OneToMany`.
- **N+1 Problem:** Dùng `@EntityGraph` trong Repository để ép Hibernate `JOIN FETCH` khi cần lấy dữ liệu phân cấp sâu.

### B. API & DTO Strategy
- **NO Entity Exposure:** Không bao giờ trả trực tiếp Entity ra Frontend. Bắt buộc map qua DTO.
- **Tree Mapping:** Sử dụng Java Streams và Đệ quy (Recursion) để map cây Entity `Criteria` sang cây DTO `CriteriaDTO`.
- **BFF Aggregation (RAM Merge):** Khi query tiến độ làm hồ sơ của sinh viên, KHÔNG dùng SQL JOIN phức tạp giữa Master Data và Transaction Data. Hãy lấy riêng rẽ, sau đó convert List Evidence thành `HashMap<CriteriaId, Evidence>` để tra cứu `O(1)` và "khâu" dữ liệu vào cây DTO ngay trên RAM bằng Java.
